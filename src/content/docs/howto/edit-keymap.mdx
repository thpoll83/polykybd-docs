This HowTo changes **what the keys do**. There are three routes, from quickest to most powerful —
pick by how permanent the change is and how deep it goes:

| Route | Best for | Recompile? | Survives a reflash? |
|---|---|---|---|
| **1. PolyKybdHost Layout Editor** | Remapping individual keys, quickly | No | No — reset to firmware defaults on reflash |
| **2. QMK keymap JSON** | A full, portable keymap you keep as a file | Yes | Yes (it's compiled in) |
| **3. QMK firmware source** | Layers, macros, tap-dance — and behaviour beyond the keymap | Yes | Yes |

Background on layers and the dynamic keymap is on [Keymaps & Layers](/using/keymaps/).

## Route 1 — PolyKybdHost Layout Editor (runtime)

The fastest route. Right-click the PolyKybdHost tray icon → **Layout Editor**, click a key, pick a
keycode, and it's written to the keyboard over HID immediately — no recompile. It writes a single
key at a time (`ID_DYNAMIC_KEYMAP_SET_KEYCODE`) and the change persists across power cycles.

Full walkthrough and limitations: [Keymap Editor](/using/keymap-editor/).

<Aside type="note">
The runtime dynamic keymap is **reset to the compiled defaults if you later reflash firmware**. For
a change that survives reflashing, use route 2 or 3.
</Aside>

## Route 2 — QMK keymap JSON (local)

A `keymap.json` is QMK's portable keymap format. It's much easier to edit than `keymap.c`, and the
local `qmk` CLI can **compile it directly**, so it's the sweet spot when you want a permanent,
shareable keymap without touching C. This is a **fully local workflow** — you edit the JSON in your
checkout and build it with your own toolchain.

<Aside type="caution">
The **online QMK Configurator ([config.qmk.fm](https://config.qmk.fm)) does not work for
PolyKybd.** It only lists keyboards that are in mainline QMK, and PolyKybd is a fork — so it won't
appear there, and you can't use it to build or download an image. Use the local `qmk` commands
below instead.
</Aside>

<Steps>

1. **Get a starting JSON** by converting the variant's existing keymap in your checkout:

   ```sh
   qmk c2json -kb polykybd/split72 -km default -o my_keymap.json
   ```

   The file holds `keyboard`, `keymap`, `layout` and a `layers` array of keycodes.

2. **Edit `my_keymap.json`** in your editor. Change keycodes, add layers — it's plain JSON. QMK's
   keycode reference is at [docs.qmk.fm/keycodes](https://docs.qmk.fm/keycodes).

3. **Compile it directly** with your local toolchain (no conversion needed), then flash per
   [Build & Flash from Source](/howto/build-and-flash/):

   ```sh
   qmk compile my_keymap.json
   ```

   Or convert it to a `keymap.c` first if you'd rather keep C sources:
   `qmk json2c my_keymap.json -o keymap.c`.

</Steps>

## Route 3 — QMK firmware source

Editing the firmware directly is the most powerful route: it does **much more than remap keys** —
new layers, macros, tap-dance, encoder behaviour, and any custom QMK logic. Use it when the JSON
format isn't enough.

The keymap *data* — the `keymaps[]` array and `encoder_map[]` — lives per-variant:

```
keyboards/polykybd/split72/keymaps/default/keymap.c
keyboards/polykybd/split42/keymaps/default/keymap.c
```

The shared keymap *logic* (rendering, HID handling, the QMK callbacks) is in
`keyboards/polykybd/poly_keymap.c`. Both are in the
[qmk_firmware fork](https://github.com/thpoll83/qmk_firmware/tree/PolyKybd); see
[Keyboard Variants](/firmware/variants/) for why the logic is shared but the data is per-variant.

<Steps>

1. **Start from a copy** so you can fall back to the default:

   ```sh
   qmk new-keymap -kb polykybd/split72 -km my_keymap    # or /split42
   ```

2. **Make the change** in `keymaps[]` (each layer is a `LAYOUT(...)` of keycodes):

   <Tabs>
     <TabItem label="Remap / add a layer">
       Replace a keycode, or add another `[n] = LAYOUT(...)` entry (up to 9 layers) plus a key that
       reaches it — `MO(n)` (momentary) or `TG(n)` (toggle).
     </TabItem>
     <TabItem label="Macro / tap-dance / custom">
       These need firmware logic, not just data — implement them via the QMK callbacks and reference
       them from `keymaps[]`. This is exactly what makes route 3 worth it. See
       [Firmware Development](/development/firmware/).
     </TabItem>
   </Tabs>

3. **Rebuild and flash** per [Build & Flash from Source](/howto/build-and-flash/):

   ```sh
   qmk compile -kb polykybd/split72 -km my_keymap
   ```

</Steps>

A keymap change lands on the per-key displays automatically — the firmware re-derives every legend
from the active layer.

See [Contributing](/development/contributing/) if you want to upstream a keymap change.
