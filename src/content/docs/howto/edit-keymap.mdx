This HowTo changes **what the keys do**. There are two routes, and which one you want depends on
whether the change needs to survive a firmware reflash:

- **Runtime** — remap individual keys over HID with the [Keymap Editor](/using/keymap-editor/). No
  recompile, instant, and stored in the keyboard's flash — but reset if you later reflash firmware.
- **Source** — edit the variant's `keymap.c`, then rebuild and flash. Permanent, and the only way to
  add layers, macros or tap-dance. That's the route this page covers.

<Aside type="note">
Background on layers and the dynamic keymap is on [Keymaps & Layers](/using/keymaps/). This page is
the step-by-step for the source route.
</Aside>

## Where the keymap lives

The shared keymap *logic* is in `keyboards/polykybd/poly_keymap.c`. Each variant's default keymap
*data* — the `keymaps[]` array and `encoder_map[]` — is in its own directory:

```
keyboards/polykybd/split72/keymaps/default/keymap.c
keyboards/polykybd/split42/keymaps/default/keymap.c
```

Both are in the [qmk_firmware fork](https://github.com/thpoll83/qmk_firmware/tree/PolyKybd)
(branch `PolyKybd`). See [Keyboard Variants](/firmware/variants/) for why the logic is shared but
the data is per-variant.

## Steps

<Steps>

1. **Start from a copy, not the default** (recommended, so you can always fall back):

   ```sh
   qmk new-keymap -kb polykybd/split72 -km my_keymap    # or /split42
   ```

   Or edit the variant's `keymaps/default/keymap.c` directly if you prefer.

2. **Make the change.** The `keymaps[]` array is data only — each layer is a `LAYOUT(...)` of
   keycodes:

   ```c
   const uint16_t PROGMEM keymaps[][MATRIX_ROWS][MATRIX_COLS] = {
       [0] = LAYOUT(
           KC_ESC,  KC_1,  KC_2,  /* ... */
       ),
       [1] = LAYOUT(
           /* function layer */
       ),
   };
   ```

   <Tabs>
     <TabItem label="Remap a key">
       Replace the keycode at that position. QMK's full keycode reference is at
       [docs.qmk.fm/keycodes](https://docs.qmk.fm/keycodes) — everything there works on PolyKybd.
     </TabItem>
     <TabItem label="Add a layer">
       Add another `[n] = LAYOUT(...)` entry (up to 9 layers) and a key that reaches it, e.g.
       `MO(n)` (momentary) or `TG(n)` (toggle).
     </TabItem>
     <TabItem label="Macro / tap-dance">
       These need firmware logic, not just data — implement them via the QMK callbacks and reference
       them from `keymaps[]`. See [Firmware Development](/development/firmware/).
     </TabItem>
   </Tabs>

3. **Rebuild and flash** your keymap. Follow [Build & Flash from Source](/howto/build-and-flash/),
   passing your keymap name:

   ```sh
   qmk compile -kb polykybd/split72 -km my_keymap
   ```

   A keymap change lands on the per-key displays automatically — the firmware re-derives every
   legend from the active layer.

</Steps>

<Aside type="tip">
For a one-off remap you don't want to recompile for, use the [Keymap Editor](/using/keymap-editor/)
instead — it writes a single key over HID (`ID_DYNAMIC_KEYMAP_SET_KEYCODE`) and takes effect
immediately. Just remember a firmware reflash resets the dynamic keymap to the compiled defaults.
</Aside>

See [Contributing](/development/contributing/) if you want to upstream a keymap change.
